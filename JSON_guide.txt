2. The Most Important Fields (For YOU)

Here are the only fields your optimizer likely needs:

✅ Food identity
"description": "Kale, raw"
"fdcId": 323505
"foodCategory": "Vegetables"

The fdcId is the unique identifier.

✅ Nutrient data (THE MOST IMPORTANT PART)

This giant section:

"foodNutrients": [ ... ]

This is an array (list) containing one object per nutrient.

Each nutrient object contains:

{
  nutrient name
  units
  measured value ("amount")
  measurement metadata
}

Example:

"Protein"
"amount": 2.92
"unitName": "g"

This means:

👉 2.92 g protein per 100 g kale
(USDA nutrients are almost always per 100 g edible portion)

3. Anatomy of One Nutrient Entry

Let’s dissect one:

{
 "nutrient": {
     "name": "Protein",
     "unitName": "g"
 },
 "amount": 2.92
}
Key pieces
Nutrient name
nutrient.name
Units
nutrient.unitName
Actual value
amount
Everything else is metadata

Example:

dataPoints
min
max
median
derivation

These describe how USDA measured or estimated the nutrient.

You can safely ignore them for optimization.

4. Why There Are TWO Energy Values

You saw:

Energy (kJ) = 148
Energy (kcal) = 35

USDA stores both.

For diet optimization:

👉 Use kcal

Ignore kJ unless you want unit conversion.

5. Why Some Nutrients Are "Calculated"

Example:

"foodNutrientDerivation": "Calculated"

This means USDA derived it from other measurements (like carb by difference).

This is normal and safe to use.

6. Portion Size Section
"foodPortions"

Example:

1 cup ≈ 20.6 g
RACC ≈ 85 g

This is super useful if you want:

User-friendly outputs

Converting grams → cups or servings

Not required for optimization, but nice bonus.

7. Nutrient Conversion Factors (Often Ignored)
"nutrientConversionFactors"

Example:

proteinValue = 2.44 kcal/g

This explains how USDA calculates calories from macros.

You usually don't need this because calorie totals are already provided.

8. inputFoods Section
"inputFoods"

This shows which samples USDA averaged together to create the final nutrient profile.

Purely scientific traceability — ignore for your project.

9. Scientific Name + Category
"scientificName"
"foodCategory"

These are excellent for filtering:

Examples:

Remove processed foods

Filter by food group

Enforce vegan constraints

10. How This Maps to Your Optimization Model

You want to transform this giant JSON object into something like:

Food	Calories	Protein	Fiber	Iron	Calcium	Sodium
Kale	35	2.92	4.1	1.6	254	53

That means extracting:

food.description
food.foodNutrients[].nutrient.name
food.foodNutrients[].amount
11. How You Would Programmatically Extract Values

Conceptually:

For each food:
    For each nutrient:
        If nutrient name matches something you care about:
            store amount

Example pseudo-Python:

for nutrient in food["foodNutrients"]:
    name = nutrient["nutrient"]["name"]
    amount = nutrient["amount"]

    if name == "Protein":
        protein = amount
12. Why JSON Uses a List Instead of Columns

CSV version would require:

protein, fiber, iron, vitamin_c, ...

But not all foods have all nutrients measured.

JSON solves this by storing nutrients dynamically.

13. Important Optimization Insight

You should NOT use nutrient names as your primary key.

Better option:

nutrient["id"]

These IDs are stable and safer for automation.

Example:

Protein ID = 1003
Energy kcal ID = 1008
14. What You Can Ignore Completely

You can safely ignore:

dataPoints

median

min/max

derivation metadata

sampling info

historical flags

publication dates

15. Minimal Data You Actually Need

For each food, extract:

fdcId
description
foodCategory
selected nutrients (amount)

That’s it.

16. Why This Structure Is Actually Very Good

It lets you:

Add/remove nutrients easily

Handle missing data

Store heterogeneous foods cleanly

It’s actually ideal for research datasets.
